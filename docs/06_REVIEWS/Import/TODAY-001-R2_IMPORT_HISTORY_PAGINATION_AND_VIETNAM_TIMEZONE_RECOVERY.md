## Root Cause

### Fixed 20-Row History

- `GET /api/import/f13/status` used `LIMIT 20` directly in the history query.
- `successCount`, `failCount`, and `lastImport` were derived from the limited row set.
- ADMIN could not review older import history or choose a larger page size.

### Vietnam Timezone Display

- `import_log.created_at` is defined as `DATETIME DEFAULT CURRENT_TIMESTAMP`.
- SQLite `CURRENT_TIMESTAMP` stores UTC in `YYYY-MM-DD HH:mm:ss` form without a timezone suffix.
- The API returned that raw string.
- The frontend used `new Date(value).toLocaleString('vi-VN')` without an explicit `Asia/Ho_Chi_Minh` display contract.

## Timestamp Contract

| Layer | Contract |
| --- | --- |
| Database | Store `import_log.created_at` as SQLite UTC-naive `YYYY-MM-DD HH:mm:ss` generated by `CURRENT_TIMESTAMP` |
| API | Serialize `created_at` as ISO 8601 UTC, for example `2026-07-16T02:54:27.000Z` |
| Frontend | Display using `Intl.DateTimeFormat('vi-VN', { timeZone: 'Asia/Ho_Chi_Minh' })` |

Old data compatibility:

- Existing `import_log` rows are consistently UTC-naive values generated by SQLite.
- No destructive migration was performed.
- API normalization appends UTC semantics only at serialization time.
- Ordering remains `ORDER BY datetime(created_at) DESC, id DESC`.

## API Contract

```text
GET /api/import/f13/status?page=1&pageSize=20
```

Allowed `pageSize` values:

- `20`
- `50`
- `100`

Invalid `page` or `pageSize` values are normalized safely.

Response includes:

- `pendingCount`
- `successCount`
- `failCount`
- `lastImport`
- `recentLogs`
- `pagination.page`
- `pagination.pageSize`
- `pagination.totalItems`
- `pagination.totalPages`
- `pagination.hasPrevious`
- `pagination.hasNext`

Summary counts are aggregate counts across the full `import_log` scope, not the current page.
