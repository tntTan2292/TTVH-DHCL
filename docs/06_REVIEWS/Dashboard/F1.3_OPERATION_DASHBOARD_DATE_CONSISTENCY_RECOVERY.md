# F1.3 Operation Dashboard Date Consistency Recovery

## Status

- Project: `QIS V2`
- Defect: `F1.3 Operation Dashboard date/data inconsistency`
- Baseline: latest `origin/codex/da-impl-006`
- Technical status: `PASS`
- PO runtime status: `AWAITING PO RECHECK`

## Root Cause

The Operation Dashboard accepted URL date filters before metadata was loaded. Invalid or future dates such as `2098-02-18` could remain visible as the selected period while downstream widgets used their own date fallback behavior.

The BCVH ranking service also resolved selected-day analysis through the latest available date inside the selected month. That allowed selected-period ranking, month-to-date values, and D-1/D-7 deltas to display data for a date different from the visible filter. When comparison rows were missing, D-1/D-7 deltas were calculated against zero instead of being reported as unavailable.

## Changed Scope

- Dashboard metadata now exposes available data coverage with `min_date` and `max_date`.
- Dashboard page normalizes invalid, out-of-coverage, and future URL dates to the available data range before issuing selected-period requests.
- Daily trend requests use the same effective `from_date` and `to_date` as the Dashboard filter.
- BCVH selected-period ranking no longer falls back to latest monthly data for a no-data selected date.
- D-1/D-7 ranking deltas are `null` when the comparison date has no data.

## Preserved Scope

- No Dashboard redesign.
- No ranking enhancements.
- No AUTO-IMPORT behavior changes.
- No TCT, HUE session, native browser-hide, or import queue changes.
